\section{Introduction}

Le projet consiste à construire un serveur de fichiers inspiré d'un FTP minimal.
Le but pédagogique n'est pas de reproduire toute la RFC FTP historique, mais de
mettre en place un protocole client/serveur cohérent, lisible et suffisamment
propre pour servir de base aux questions suivantes du sujet.

\begin{infobox}
Le dépôt courant réalise complètement l'étape I du sujet :
\textbf{les questions Q1 à Q7 sont terminées}. Le client fonctionne maintenant sous
la forme \texttt{./clientFTP <host>}, affiche l'invite \texttt{ftp>}, accepte
\texttt{get <nom\_fichier>}, reçoit le fichier et affiche les statistiques de
transfert demandées.
\end{infobox}

L'objectif de ce compte-rendu est double :

\begin{itemize}[leftmargin=*]
    \item expliquer l'architecture du projet ;
    \item expliquer pourquoi le code fonctionne, en particulier du point de vue réseau.
\end{itemize}

\newpage \section{Architecture générale du projet}

\subsection{Principe d'organisation}

Le projet est structuré en couches courtes et spécialisées. Cette séparation est
importante : elle évite de concentrer toute la logique réseau, toute la logique
métier et toute la logique d'IHM dans un seul fichier.

\begin{tipbox}
Une bonne architecture ici consiste à faire en sorte que :

\begin{itemize}[leftmargin=*]
    \item le \texttt{main()} du client s'occupe surtout de l'interaction utilisateur ;
    \item le \texttt{main()} du serveur s'occupe surtout de l'écoute et des workers ;
    \item les fichiers \texttt{client\_requests.c} et \texttt{server\_requests.c}
    portent la logique métier ;
    \item la sérialisation et les structures communes restent dans la couche protocole.
\end{itemize}
\end{tipbox}

\newpage \section{Client et serveur : deux machines différentes}

\begin{warningbox}
Même si les tests locaux se font souvent sur une seule machine, il faut raisonner
comme si le client et le serveur étaient sur deux machines différentes.

Autrement dit :

\begin{itemize}[leftmargin=*]
    \item \texttt{data\_server/} représente le disque local du serveur ;
    \item \texttt{data\_client/} représente le disque local du client ;
    \item le fichier ne doit jamais être copié par accès direct au système de
    fichiers ;
    \item le fichier doit obligatoirement passer par la socket TCP.
\end{itemize}
\end{warningbox}

Cette manière de raisonner explique plusieurs choix du projet :

\begin{itemize}[leftmargin=*]
    \item le client ne lit jamais dans \texttt{data\_server/} ;
    \item le serveur n'écrit jamais dans \texttt{data\_client/} ;
    \item la structure \texttt{request\_t} transporte le nom du fichier via le réseau ;
    \item la structure \texttt{response\_t} transporte le statut et la taille du fichier ;
    \item le contenu du fichier est envoyé comme un flux d'octets TCP.
\end{itemize}

\newpage \section{Description du protocole}

\subsection{Constantes et structures communes}

Le fichier \texttt{include/ftp\_shared.h} définit les éléments communs au client
et au serveur.

\begin{itemize}[leftmargin=*]
    \item \texttt{FTP\_PORT} fixe le port d'écoute à \texttt{2121}.
    \item \texttt{FTP\_PROTO\_VERSION} identifie la version courante du protocole.
    \item \texttt{FTP\_BLOCK\_SIZE} fixe la taille du tampon utilisé pour recevoir
    un payload côté client.
    \item \texttt{NB\_PROC} fixe le nombre de processus workers du serveur.
\end{itemize}

Deux structures sont au cœur du dialogue :

\begin{itemize}[leftmargin=*]
    \item \texttt{request\_t} : message client vers serveur ;
    \item \texttt{response\_t} : message serveur vers client.
\end{itemize}

\subsection{Pourquoi les structures sont utiles}

\begin{sciencebox}
Utiliser des structures fixes simplifie beaucoup la logique :

\begin{itemize}[leftmargin=*]
    \item le client sait exactement quoi envoyer ;
    \item le serveur sait exactement combien d'octets lire pour une requête ;
    \item le client sait exactement combien d'octets lire pour une réponse ;
    \item le champ \texttt{file\_size} permet ensuite de savoir combien d'octets
    lire pour le contenu du fichier.
\end{itemize}
\end{sciencebox}

Le type \texttt{typereq\_t} est également pensé pour l'avenir. Même si seule la
commande \texttt{GET} est exploitée aujourd'hui, les types futurs \texttt{PUT},
\texttt{LS}, \texttt{RM}, \texttt{BYE} et \texttt{AUTH} existent déjà pour garder
une base extensible.

\newpage \section{Comment ça marche côté réseau}

\subsection{Pourquoi TCP convient au projet}

Le projet repose sur TCP. C'est un choix naturel pour un mini-FTP, car TCP fournit :

\begin{itemize}[leftmargin=*]
    \item une livraison fiable des octets ;
    \item le respect de l'ordre d'envoi ;
    \item la détection de la fermeture de la connexion ;
    \item une abstraction pratique pour transporter aussi bien des structures
    de protocole que des fichiers binaires.
\end{itemize}

\begin{sciencebox}
TCP est un \textbf{flux d'octets} et non un système de messages déjà découpés.
Cela signifie qu'un \texttt{read()} peut renvoyer moins d'octets que demandé.
Le projet s'appuie donc sur \texttt{Rio\_readn()} et \texttt{Rio\_writen()}
pour garantir l'envoi et la réception complets des messages attendus.
\end{sciencebox}

\subsection{Ouverture de la connexion}

\subsubsection*{Côté serveur}

Le serveur appelle \texttt{Open\_listenfd(FTP\_PORT)}. Dans \texttt{csapp}, cela
effectue les étapes classiques :

\begin{enumerate}[leftmargin=*]
    \item création d'une socket IPv4 de type \texttt{SOCK\_STREAM} ;
    \item activation de \texttt{SO\_REUSEADDR} ;
    \item association à \texttt{INADDR\_ANY} ;
    \item conversion du port avec \texttt{htons()} ;
    \item \texttt{bind()} puis \texttt{listen()}.
\end{enumerate}

\subsubsection*{Côté client}

Le client appelle \texttt{Open\_clientfd(host, FTP\_PORT)}. Cette fonction :

\begin{enumerate}[leftmargin=*]
    \item résout le nom d'hôte avec \texttt{getaddrinfo()} ;
    \item récupère l'adresse IPv4 du serveur ;
    \item convertit le port avec \texttt{htons()} ;
    \item lance \texttt{connect()}.
\end{enumerate}

\subsection{Pourquoi les fonctions \texttt{hton}/\texttt{ntoh} sont importantes}

\begin{warningbox}
Deux machines différentes peuvent ne pas représenter les entiers de la même façon
en mémoire. Il faut donc distinguer l'ordre hôte et l'ordre réseau.

Les conversions classiques sont :

\begin{itemize}[leftmargin=*]
    \item \texttt{htons()} et \texttt{ntohs()} pour les entiers 16 bits ;
    \item \texttt{htonl()} et \texttt{ntohl()} pour les entiers 32 bits.
\end{itemize}
\end{warningbox}

Dans le projet actuel, ces conversions sont déjà bien utilisées au niveau des
sockets, notamment pour les numéros de port.

En revanche, il faut être précis sur le protocole applicatif :

\begin{itemize}[leftmargin=*]
    \item les structures \texttt{request\_t} et \texttt{response\_t} sont encore
    envoyées telles quelles ;
    \item cela fonctionne dans nos tests, car client et serveur sont compilés dans
    un environnement homogène ;
    \item pour une vraie robustesse inter-machines, il faudrait sérialiser
    explicitement tous les champs numériques du protocole.
\end{itemize}

\begin{tipbox}
La prochaine amélioration réseau réellement propre serait d'introduire une
conversion d'ordre des octets sur tous les champs entiers de \texttt{request\_t}
et \texttt{response\_t}, y compris les tailles de fichier et offsets.
\end{tipbox}

\newpage \section{Déroulement complet d'une requête \texttt{GET}}

\subsection{Démarrage du serveur}

\begin{enumerate}[leftmargin=*]
    \item \texttt{ftp\_server\_run()} appelle \texttt{ftp\_enter\_working\_directory()} et se place dans \texttt{data\_server/}.
    \item Le serveur ouvre la socket d'écoute sur le port \texttt{2121}.
    \item Le père crée \texttt{NB\_PROC} workers avec \texttt{Fork()}.
    \item Chaque worker entre dans \texttt{worker\_loop()} et attend une connexion.
\end{enumerate}

\subsection{Démarrage du client}

\begin{enumerate}[leftmargin=*]
    \item \texttt{ftp\_client\_run(host)} se place dans \texttt{data\_client/}.
    \item Le client ouvre une connexion TCP vers l'hôte demandé.
    \item Il affiche \texttt{Connected to ...} puis \texttt{ftp> }.
    \item Il lit une ligne utilisateur avec \texttt{Fgets()}.
\end{enumerate}

\subsection{Parsing et construction de la requête}

La fonction \texttt{parse\_get\_command()} extrait le nom du fichier et vérifie
que la commande est bien de la forme \texttt{get <nom\_fichier>}.

Ensuite, \texttt{ftp\_client\_get()} :

\begin{enumerate}[leftmargin=*]
    \item vérifie que le nom de fichier est sûr ;
    \item initialise \texttt{request\_t} via \texttt{ftp\_build\_get\_request()} ;
    \item mesure le temps de transfert ;
    \item envoie la requête au serveur avec \texttt{Rio\_writen()}.
\end{enumerate}

\subsection{Traitement côté serveur}

Le worker serveur reçoit la connexion puis :

\begin{enumerate}[leftmargin=*]
    \item lit une \texttt{request\_t} complète avec \texttt{ftp\_receive\_request()} ;
    \item vérifie version, type et nom de fichier avec \texttt{validate\_request()} ;
    \item traite le \texttt{GET} avec \texttt{handle\_get\_request()} ;
    \item charge le fichier demandé avec \texttt{ftp\_load\_file()} ;
    \item envoie \texttt{response\_t} ;
    \item envoie le contenu binaire du fichier.
\end{enumerate}

\subsection{Réception côté client}

Le client :

\begin{enumerate}[leftmargin=*]
    \item lit \texttt{response\_t} avec \texttt{ftp\_receive\_response()} ;
    \item vérifie le statut ;
    \item si le statut est \texttt{OK}, lit exactement \texttt{file\_size} octets ;
    \item écrit ces octets dans un fichier local avec \texttt{ftp\_receive\_file\_payload()} ;
    \item calcule la durée et le débit moyen ;
    \item affiche le message final.
\end{enumerate}

\begin{examplebox}
\textbf{Chemin logique d'un transfert réussi}

\begin{center}
\texttt{clientFTP} $\rightarrow$ \texttt{request\_t} $\rightarrow$ \texttt{serverFTP}

\texttt{serverFTP} $\rightarrow$ \texttt{response\_t + payload} $\rightarrow$ \texttt{clientFTP}
\end{center}
\end{examplebox}

\newpage \section{Pourquoi l'implémentation fonctionne}

\subsection{Parce qu'une connexion transporte une seule requête}

Le projet suit une règle simple :

\begin{quote}
\textbf{Une seule requête par connexion TCP.}
\end{quote}

Cette décision simplifie fortement le protocole :

\begin{itemize}[leftmargin=*]
    \item le client ouvre une connexion ;
    \item il envoie une requête ;
    \item il reçoit une réponse ;
    \item il reçoit éventuellement un fichier ;
    \item la connexion est ensuite fermée.
\end{itemize}

Il n'y a donc pas, à ce stade, de boucle de commandes persistante côté serveur.

\subsection{Parce que la taille des échanges est connue}

Le serveur et le client savent toujours combien d'octets attendre :

\begin{itemize}[leftmargin=*]
    \item une requête a toujours la taille de \texttt{request\_t} ;
    \item une réponse a toujours la taille de \texttt{response\_t} ;
    \item un fichier a la taille \texttt{file\_size} annoncée dans la réponse.
\end{itemize}

Cette propriété évite les ambiguïtés dans le flux TCP.

\subsection{Parce que la sécurité minimale est centralisée}

La fonction \texttt{ftp\_is\_safe\_filename()} bloque les demandes dangereuses,
comme :

\begin{itemize}[leftmargin=*]
    \item \texttt{get ../secret} ;
    \item \texttt{get /etc/passwd} ;
    \item \texttt{get .} ;
    \item \texttt{get ..}.
\end{itemize}

Le contrôle est volontairement centralisé pour que client et serveur reposent sur
la même règle de validation.

\subsection{Parce que l'arrêt du serveur est propre}

Le serveur gère correctement l'arrêt via \texttt{SIGINT} :

\begin{itemize}[leftmargin=*]
    \item le père relaie le signal aux fils ;
    \item les descripteurs utiles sont fermés ;
    \item les workers sortent proprement ;
    \item le père attend leur terminaison.
\end{itemize}

Cette logique évite les processus zombies et les ressources laissées ouvertes.

\newpage \section{Limites actuelles}

Même si l'étape I est terminée, plusieurs limites sont normales à ce stade :

\begin{itemize}[leftmargin=*]
    \item une seule requête par connexion ;
    \item chargement complet du fichier en mémoire côté serveur ;
    \item pas encore de reprise sur panne ;
    \item pas de \texttt{put}, \texttt{ls}, \texttt{rm} ;
    \item pas d'authentification ;
    \item sérialisation applicative encore simplifiée.
\end{itemize}

\begin{sciencebox}
La vraie amélioration réseau à moyen terme n'est pas seulement d'ajouter des
commandes. Il faudra aussi rendre le protocole plus portable entre machines
hétérogènes en sérialisant explicitement les champs numériques avec les conversions
d'ordre des octets adaptées.
\end{sciencebox}

\newpage \section{Conclusion}

Le projet obtenu est un mini-FTP pédagogique cohérent et correctement structuré.
La partie demandée par l'étape I du sujet est terminée jusqu'à Q7 incluse.

Sur le plan technique, ce projet montre :

\begin{itemize}[leftmargin=*]
    \item comment organiser un client et un serveur en C ;
    \item comment séparer interface, logique métier, protocole et transfert ;
    \item comment utiliser des sockets TCP proprement ;
    \item comment fiabiliser les échanges avec \texttt{Rio\_readn()} et \texttt{Rio\_writen()} ;
    \item pourquoi les conversions \texttt{htons}, \texttt{htonl}, \texttt{ntohs}, \texttt{ntohl} comptent dès que l'on pense réellement à deux machines distinctes.
\end{itemize}

La suite logique du travail serait :

\begin{enumerate}[leftmargin=*]
    \item sérialiser complètement \texttt{request\_t} et \texttt{response\_t} ;
    \item passer à un transfert par blocs ;
    \item autoriser plusieurs requêtes par connexion ;
    \item implémenter les commandes avancées et l'authentification demandées dans la suite du sujet.
\end{enumerate}
